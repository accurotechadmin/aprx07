# Guide for Updating Secondary Full Boot Options

## Purpose

This guide explains how to update every secondary boot option so each option benefits from the SVG boot evaluation while still preserving its own domain-specific strengths. It is intentionally forward-looking: it applies lessons that future option-by-option reviews are likely to uncover, so maintainers can improve the whole secondary menu before each option has been individually audited.

## Core lesson from the SVG option

The SVG option works best when it does four things early:

1. Displays its featured mode-specific optimizations above the fold.
2. Separates boot selection from file-changing actions.
3. Gives a concrete workflow for the next actionable user request.
4. States validation, citation, and final-response expectations before artifact work begins.

Every other secondary boot option should adopt the same pattern, with its own equivalent of “math skeleton before style.”

## Standard update pattern for every secondary boot file

Each secondary boot document should contain these sections near the top, in this order where practical:

1. **Purpose** — the option's job in one or two paragraphs.
2. **When to use** — clear routing boundaries and combination triggers.
3. **Required reading order** — smallest useful source set.
4. **Above-the-fold boot display** — primitive UI box with a `:: FEATURE ::` line.
5. **Operating rules** — domain-specific guardrails.
6. **Actionable request workflow** — what happens after the user gives a real task.
7. **Boot-time actions** — action table with effect, default output, and edit-risk label.
8. **Validation or quality checks** — commands, review passes, or source checks.
9. **MVP response contract** — what the assistant says after loading the path.
10. **First-pass checklist** — final pre-output guardrail.

Do not force artificial uniformity when a domain needs different vocabulary, but keep the same functional components.

## Primitive UI pattern

Use this pattern for each option's above-the-fold display:

```text
╔════════════════════════════════════════════════════════════════╗
║ FIBONACCI EXPLORER // <BOOT NAME>                            ║
║ <MODE LABEL>: READY                                          ║
╚════════════════════════════════════════════════════════════════╝
> :: FEATURE :: <option-specific-optimization-a> | <b> | <c>
> :: FAST PATH :: option number loads now; actions may be added next
> :: CONFIRM PATH :: use actions/info/combine to preview before finalizing

   /\
  /**\
 /****\
   ||

> No artifact has been created yet; awaiting actionable task.
```

The tree is decorative. Use zero to three trees per screen view. Never use a tree as a status icon, warning, source marker, data mark, or navigation control.

## General refinements to bake in across the board

### Fast path and confirm path

All options should support both fast and confirm behavior:

- A plain number or exact short name loads immediately.
- `actions <boot>`, `info <boot>`, `combine <a> + <b>`, or ambiguous input previews before finalizing.
- After fast-path loading, the assistant should say actions may still be added on the next prompt.

### Edit-risk labels

Every boot action should be labeled:

- **Read-only** for inspection and summarization.
- **Draft-only** for proposed content not yet saved.
- **File-modifying** for actions that can create or edit files after a concrete request.

Boot-time selection alone should not modify files.

### Actionable workflow

Every option should define the next-task sequence: parse intent, declare contract, inspect sources/assets, build factual skeleton, produce output, validate, summarize.

### Source inspection before generation

Each option should identify what it must inspect before creating anything:

- SVG option: existing drawings.
- Document/index options: existing link indexes and document-relation blocks.
- Claim options: statement index, inventories, and cohesion notes.
- Data options: workbook, formulas, column definitions, and range declarations.
- Teaching options: learner misconceptions and staged concept order.

### Validation expectations

Validation should be domain-specific but explicit. Examples:

- XML parsing for SVG files.
- `git diff --check` for documentation edits.
- Link-target checks for index updates.
- Formula recomputation for spreadsheet/data work.
- Citation/source-status checks for claims.
- Reading-order and misconception checks for lessons.

## Recommended update procedure

1. Update the global secondary menu first so every option has shared vocabulary for fast path, confirm path, edit-risk labels, compact menus, and featured lines.
2. Update the SVG option as the reference implementation.
3. For each remaining option, add an above-the-fold display, actionable workflow, action edit-risk table, validation expectations, and first-pass checklist.
4. Run documentation checks after each batch.
5. Avoid changing source authority or mathematical claims while improving boot ergonomics.

## Definition of done for a secondary boot option

A secondary boot option is maintenance-ready when:

- the user can choose it by number or short name;
- the assistant knows whether to fast-load or preview actions;
- its featured optimizations are visible above the fold;
- its actions have edit-risk labels;
- it defines what happens after an actionable task;
- it names what must be inspected before creating new output;
- it defines validation or quality checks;
- it preserves repository source hierarchy and Fv convention discipline;
- it can combine with at most two adjacent modes without confusion.
