## What this is

This repository is a Fibonacci knowledge, audit, graphing, and creative-exploration system. It is not just a folder of notes. It is a working canon for making Fibonacci claims precise, making visualizations honest, and helping humans and expert coding LLM sessions move between mathematics, art, documents, diagrams, data, metaphor, and empirical caution without blurring their truth-status.

The repository is built around a shared convention:

- `Fv -1 -> destination 0`
- `Fv 0 -> destination 1`
- `Fv 1 -> destination 1`
- later destinations are produced by adding the two previous destinations.

That convention is used to protect the difference between a process step, a destination value, a conventional Fibonacci index, a transition, a ratio, a finite approximation, an asymptotic limit, a rendered graphic, and an interpretive metaphor.

Use this repository when you want to:

- create Fibonacci-accurate explanations;
- audit charts, diagrams, or claims;
- draw mathematically disciplined SVGs and visual proofs;
- explore golden-ratio convergence without false exactness;
- design art that respects the underlying structure;
- build document indexes and cross-reference maps;
- separate exact mathematics from empirical resemblance and metaphor;
- turn small Fibonacci insights into larger families of prompts, figures, workbooks, manifests, and lessons.

## How to boot a new Codex session

Start a new Codex session at the root of this repository and type:

```text
boot
```

The session should find and read [boot.md](boot.md). That primary boot equips it with the source hierarchy, Fv convention, canonical destination table, exactness rules, graphing discipline, geometry warnings, evidence boundaries, and secondary boot routing.

After primary boot, the session now opens an interactive secondary boot menu automatically. The menu uses the repository's friendly plain-text CRT style consistently across the primary boot, secondary menu, and individual secondary boot displays. Choose a numbered option, type a short boot name, describe the task, or add boot-time secondary actions with a plus sign:

```text
3 + sketch-plan, label-audit
```

or:

```text
finish booting with document-indexer.boot.md + link-audit
```

The available secondary boot options, secondary actions, and UI controls are listed in [secondary_boot_prompts.md](secondary_boot_prompts.md). You can also describe your task and let the session choose the smallest relevant secondary boot path:

```text
finish booting with whatever secondary boot path is best for auditing a shell spiral claim
```

Recommended workflow:

1. **Primary boot:** type `boot` and let the session read [boot.md](boot.md).
2. **Interactive secondary menu:** choose a path from [secondary_boot_prompts.md](secondary_boot_prompts.md), choose optional boot-time actions, or describe the task.
3. **Task prompt:** ask for the actual work only after the session confirms that both boot layers and any selected actions are complete.

## Hidden handrails, creative accelerators, and things most READMEs forget

### 1. The zero is an address, not a decorative preface

The Fv -1 destination `0` is not there merely to make the sequence look familiar. It anchors the repository's address system. If a graph, table, or explanation silently drops it, every later convention comparison becomes easier to misunderstand.

### 2. The duplicate one is a hinge

Fv 0 and Fv 1 both have destination `1`, but they are not the same event. Many indexing mistakes begin when equal value is mistaken for equal process identity.

### 3. Ratios live on edges, not just nodes

A ratio uses an adjacent pair. When teaching or drawing ratios, put the ratio concept on the transition between destinations, not only on one endpoint. This keeps numerator, denominator, step, and direction visible.

### 4. A graph should answer a question, not just display a sequence

Before drawing, ask: what will the viewer discover here that they would not discover as easily from a table? If the answer is unclear, the graph may be decoration rather than insight.

### 5. Smoothness is suspicious

Fibonacci destinations are integer-step observations. Lines may show order, but smooth splines can invent false intermediate states, false crossings, false extrema, and false continuity.

### 6. `phi` is a limit, not a late-stage trophy value

No finite Fibonacci forward ratio equals `phi` exactly. Late ratios can be close enough that rounding hides which side of `phi` they are on. Keep exact fractions, rounded decimals, and limits separate.

### 7. The prettiest spiral is often the most dangerous claim

A Fibonacci square spiral, a golden rectangle, and an exact logarithmic golden spiral are related ideas, not interchangeable objects. Artistic resemblance should invite closer labeling, not looser language.

### 8. Screenshots are witnesses, not authorities

Pixels have line widths, antialiasing, export scaling, compression, and cropping. Source data and coordinate definitions outrank screenshot measurements.

### 9. Metaphor is welcome when it wears a badge

The repository supports metaphors such as Yesterday = `1/phi`, Today = `1`, Tomorrow = `phi`. The rule is not “avoid metaphor.” The rule is “label metaphor as metaphor and define what is being normalized.”

### 10. Empirical Fibonacci claims need rivals

For shells, plants, markets, bodies, architecture, or behavior, do not ask only “Can Fibonacci be seen here?” Ask what non-Fibonacci model would also fit, what residuals look like, what sample size exists, and what evidence would distinguish resemblance from mechanism.

### 11. Expansion should return to exactness

Creative branches are encouraged. The best branches return to the exact core with better language, better diagrams, better evidence boundaries, or better questions.

### 12. The repository can be used like an instrument

Think of [boot.md](boot.md) as tuning, [secondary_boot_prompts.md](secondary_boot_prompts.md) as choosing the instrument section, [the_essences.md](the_essences.md) as the score's key signature, [draw.md](draw.md) as technique, and [insightgraphsuite.md](insightgraphsuite.md) as improvisational vocabulary.

### 13. A future catalogue should index relationships, not just filenames

A useful master catalogue should record what each document governs, which documents it depends on, which documents depend on it, what claims it stabilizes, what outputs it enables, and what cautions it carries.

### 14. The best prompt is often a two-stage prompt

First boot the shared canon. Then finish booting into the specific mode. This prevents every task from re-solving the entire repository while still keeping the work rooted in the same truth system.
